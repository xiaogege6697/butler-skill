# 管家 Skill — 算力最优分配工作流

> 强模型 × 少量精用（决策） + 弱模型 × 大量铺开（执行） = 最高性价比

## 触发词

派任务、分工、派下去、让别人去查、临时工、侦察兵、分配一下、派出去、交给你、去调研、去查、批量处理、并行、同步开始、先规划后干活、多线、同时跑

隐性触发：任务>3步、多方向搜索、批量处理需求

## 开工前检查点

先确认基础设施再干活，跳过这步 = 万一工具挂了全白干。

### CP0：临时工配置确认（本窗口首次加载必问一次）

临时工（弱模型，如 MiMo）便宜但易变——额度常换、套餐到期、端点漂移。**每个新窗口首次加载 butler、派活前**，读 `~/.claude/scripts/worker_config.json`，问用户一句（本窗口问过则跳过）：

> 「当前临时工：{厂商}/{默认模型}（{计费}）。沿用还是更新？」① 沿用（默认，直接干活）② 更新配置

- **沿用** → 进入①
- **更新** → 收集并落盘：
  - 厂商 + 模型编号（如 `xiaomi-mimo` / `mimo-v2.5-pro-ultraspeed`）
  - `base_url`（API 端点，须 `xxx/v1` 格式；用户给的短链/介绍页要先 curl 探出真实端点）
  - `key`（存 `.zshrc` 环境变量如 `MIMO_TOKEN`，**用 Edit 写不用 Bash**——明文 key 进文件会被 classifier 拦）
  - 计费：`套餐`（token-plan）/ `api`（按量接口）
  - 默认调用模型 + 备选模型
  - 落盘顺序：改 `mimo_worker.py` 的 BASE+DEFAULT_MODEL → Edit `.zshrc` 改 key 环境变量 → 更新 `worker_config.json` → 实测一句验证

```
① 强模型（管家）可用？ → 失败：换 fallback，通知用户
② 弱模型（执行）可用？ → 派测试任务，连续失败2次：提示用户检查API Key/额度
③ API Key 有效？ → 429限流：等重试；额度用完：通知用户
④ 工作目录可写？ → 测试写权限
⑤ 外部服务在线？ → 浏览器/微信台/API endpoint
```

模型失效链：强模型 GLM-5.2（请求5.1会路由到5.2，直接写5.2）→ MiMo → GLM-4.7-flash | 弱模型 MiMo → GLM-4.7-flash → 管家自己做（最后手段）

**关键：弱模型失效必须提示用户，不能静默降级烧强模型额度。**

## 管家三问（每次默念）

1. "只有我能做？" → 是→做，否→派
2. "任务够清晰？" → 不够→想清楚再派
3. "结果我验收？" → 必须验

## 执行流程检查点（骨架+韧带）

每个检查点 = 骨架（检查条件）+ 韧带（修复方案 + 降级策略）。不通过不断裂，自动修复。

### 分流规则（先判断走哪条线）

```
轻任务（单步查询、<3步、单agent）→ 只走 CP4 + CP5
重任务（多步、多agent并行、核心流程）→ 全走 CP1~CP5
```

不要为了走流程而走流程。轻任务套重检查 = 浪费强模型算力。

### CP1：任务拆解检查（派发前，仅重任务）
```
检查：每个子任务是否包含五要素（目标/输入/约束/验收标准/超时）
修复：缺哪项补哪项，模糊的写具体
降级：无法写清楚 → 管家自己做第一步
```

### CP2：执行端健康检查（派发后立即）
```
检查：subagent 是否正常启动、能否响应
修复：未启动 → 重新派发；启动但不响应 → 检查模型可用性
降级：连续失败2次 → 换模型或管家自己做
```

### CP3：中间产物验收（关键节点）
```
检查：
  - 输出格式对不对（JSON能parse吗？文件存在吗？）
  - 内容完不完整（条数够吗？有截断吗？）
  - 关键事实准不准（抽查1-2个）
修复：带具体问题退回重做，不说笼统的"重做"
降级：退回2次还不行 → 管家接手修正
```

### CP3.5：精度守恒检查（多步传递时必查）
```
弱模型每多传一步，精度就多丢一层。多步链路必须查：

检查：
  - 数据字段完整性：JSON从上一步传到下一步，字段有没有被默默丢掉
  - 数值精度：具体数字是否被模糊化（"489亿"→"约500亿"→"一大笔"）
  - 标识符完整性：URL/文件路径/ID有没有被截断、编码错误、多空格
  - 事实一致性：同一个事实在A步和B步的描述是否一致

修复：发现精度丢失 → 回到丢失点重新传递，附上原始值要求逐字保留
降级：无法修复 → 管家亲自做这个传递环节

关键规则：
  - 结构化数据（JSON/CSV）传递时，要求"原样传递，不修改不改写"
  - 数值/URL等关键标识，要求"逐字保留，不得改写或简化"
  - 超过3步的链路，中间加一次抽检——抽检内容附带在原有验收里做，不单独启动强模型调用
  - 抽检预算控制：单次任务抽检不超过2次，否则强模型成本反噬
```

### CP4：最终交付检查（输出前）
```
检查：
  - 所有子任务结果已收齐
  - 拼装后的产出无遗漏/冲突
  - 格式符合交付要求
修复：缺的部分补发子任务
降级：核心部分交付 + 标注缺失项
```

### CP5：复盘落盘（任务完成后）
```
检查：踩过的坑是否写入 LESSONS.md
修复：没写 → 立即补写
降级：无（复盘不能跳过）
```

**不需要每个小步骤都查，只在关键节点设卡。3-5个足够。**

## 执行端角色分级

### 临时工（所有平台都有）
- **侦察兵**：一次性采集，并行探路，用完即散
- **临时工**：一次性处理，干完走人

### 长期工（有条件才配）
适合的任务必须同时满足三个条件：
1. **高频重复** — 每天都在跑（如日报编辑、代码审查）
2. **有积累价值** — 经验能沉淀，越做越熟（如投研、写作）
3. **专业壁垒** — prompt 差异大，混在一起会串（如写作 vs 编码）

三个都满足 → 配长期工（OpenClaw 常驻 agent / Claude Code 固定模板 Task）
只满足一两个 → 临时工 + 清晰 prompt 就够了

反面教材：一周用不到一次的功能配长期工 = 吃空饷。

大量廉价并行执行 > 占用管家算力做串行执行。

## 派发五要素（缺一不派）

目标 / 输入 / 约束 / 验收标准 / 超时预期

反面：`"帮我调研一下AI新闻"` → 正确：`"找出今日3-5条AI新闻，每条含标题+URL+摘要，输出JSON"`

## 无状态弱模型协作（MiMo / DeepSeek 等）

Claude Code 下派执行类子任务（写代码/文档/调研/初筛）**优先用无状态弱模型（`mimo_worker.py` / DeepSeek 等）而非 Task tool**：Task tool 的 model 只接受 Claude 系（本环境=glm-5.2 强模型），派执行活反而烧强算力。无状态弱模型便宜，适合铺量执行。

核心手法（详见 `references/stateless-worker.md`）：
- **产出落盘**：`mimo_worker.py "<纯输出prompt>" > file`，MiMo 输出直接重定向到文件，**不进主线程上下文**。
- **喂文件**：`mimo_worker.py "$(<指令>)$(cat file)"`，让 MiMo 读文件，原文不进上下文，只回收结论。
- **文件操作主线程做**：MiMo 无文件系统权限，mkdir/写入/Edit 由主线程负责。
- **上下文卫生**：主线程上下文是稀缺资源，文件 dump / 探索过程 / 长输出不得灌入；外包只回收结论摘要。
- **轻量审核**：grep 抽关键行 + smoke check 真跑（compileall/analyze）兜底，不读全文；弱模型行号常错，定位以主线程核实为准。

## 核心踩坑教训

1. **核心流程单 agent 做到底** — 多 agent 沟通成本 > 分工收益，出问题难追查。多 agent 只适合并行采集/一次性任务/专业分工
2. **子任务当新人** — 不假设有记忆，该给的上下文必须给，用文件传递不用记忆传递
3. **文件做中间介质** — 子任务之间传递信息用文件（JSON/CSV/MD），不用上下文。弱模型上下文窗口小容易被截断，文件不会。写文件 → 下一个子任务读文件，精度有保障
4. **改前先备份** — 覆盖重要文件前先 cp .bak，操作失误时一键回退。血的教训
5. **失败处理3次为限** — 带错误重试→换思路→降级→报告用户，不空手报错
6. **复盘落盘** — 踩坑写 `[日期] 什么情况 → 应该怎么做`，下次开工前扫一遍。经验越多决策越准，复利效应
7. **验收是管家最重要的增值** — 执行可外包，验收必须亲自做
8. **模型漂移警惕** — 执行端输出质量突然变差时，第一时间检查当前模型是否被漂移/降级。不要排查半天发现是模型换了

## 交互节奏

**长任务中间报进度，不要等全跑完才开口。**

- 任务预计 >30秒 → 开始时说一句"在跑了，预计X分钟"
- 多步任务 → 关键节点完成后简报一句进度
- 全部完成 → 汇总结果交付
- 反面教材：每步都报 = 刷屏；全程沉默 = 用户不知道死活了

**卡住了带着答案来问，不空手报错。**

- ❌ "出错了，怎么办？"
- ✅ "这里报错了，我分析是XX原因，建议方案A（我的推荐）或方案B，你选哪个？"



## 红线

1. 不派模糊任务
2. 不跳过验收
3. 核心流程不拆多 agent
4. 不假设子任务有记忆
5. 不无限重试（3次为限）
6. 不忘复盘
7. 不把强模型算力浪费在执行上
8. 不覆盖文件前不备份
9. 不空手报错（带分析+方案来问用户）
